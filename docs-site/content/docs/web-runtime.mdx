Cinder's browser support is a terminal backend, not a React recreation of a terminal application.

```text
Dart application
  → Widget tree
  → Element reconciliation
  → RenderObject layout and paint
  → terminal cell buffer
  → differential ANSI output
  → WebBackend
  → window.cinderBridge
  → xterm.js
```

The application still uses Cinder's scheduler, focus tree, hit testing, layout, painting, and buffer diff. Next.js owns the surrounding documentation page and isolated runner frame. xterm.js owns terminal emulation and the browser DOM surface.

## Runtime isolation

Every `/play/[slug]/` route runs in its own document context. A restart reloads that document instead of trying to reset framework singletons in place.

The host starts in this order:

1. create xterm.js and the fit addon;
2. attach the terminal to a non-zero viewport;
3. publish `window.cinderBridge` with the initial column and row count;
4. subscribe to text, binary, and resize events;
5. load the compiled Dart guest bundle;
6. let `WebBackend` register input, resize, output, and shutdown callbacks;
7. focus the terminal after the guest has loaded.

Teardown reverses ownership:

1. mark the host disposed;
2. cancel scheduled fitting;
3. disconnect `ResizeObserver`;
4. dispose xterm subscriptions;
5. invoke the guest shutdown callback;
6. remove the guest script;
7. dispose xterm.js;
8. remove `window.cinderBridge`;
9. destroy the isolated frame on navigation or restart.

This prevents previous timers, focus managers, stream controllers, and terminal bindings from leaking into the next application.

## Example runtime modes

The generated example manifest assigns one explicit mode to every official source.

### Direct web

`direct-web` means the original repository Dart source compiles for the browser and runs without a capability replacement.

The detail page links to the original source and loads its generated bundle directly.

### Browser adapter

`browser-adapter` means the Cinder UI and educational intent remain, but a native data source is replaced by a real browser-compatible implementation.

Examples include:

- generated in-memory RGBA pixels instead of an arbitrary local image path;
- an in-memory log ledger instead of writing a debug file;
- browser resize constraints instead of reading terminal dimensions from `dart:io`.

The page links to both the original source and the adapter source.

### Browser sandbox

`browser-sandbox` is used when the native operation itself cannot exist in a static browser site.

A sandbox must be deterministic and visibly labelled. It may demonstrate UI, input, state transitions, and protocol behavior, but it must not claim to be:

- a real PTY or operating-system shell;
- an SSH connection;
- unrestricted process execution;
- arbitrary host filesystem access;
- a native FFmpeg process;
- raw sockets or FFI.

### Native only

`native-only` keeps the source and route visible while explaining the capability that requires a native runtime.

The route is not silently omitted, and the site does not display a fake success state.

### Build failed

`build-failed` means discovery succeeded but isolated `dart compile js` failed. CI stores compiler diagnostics and the route presents the failure reason.

This is different from `native-only`: a build failure may be a portable bug worth fixing, while a native boundary can be intentional.

## Package-scoped examples

Examples under `packages/<package>/example/` are compiled with that package's own dependency graph instead of the root package configuration.

The recovery pass:

1. identifies the containing package from the repository path;
2. resolves that package with `dart pub get` only when an isolated build is needed;
3. creates a temporary launcher inside the package root;
4. compiles from that package root so package imports and dependency overrides remain authoritative;
5. writes the resulting static bundle into the shared generated example directory.

This prevents a valid integration example from being marked as failed merely because its package import is unavailable from the root package configuration.

## Input and resize

The browser host forwards:

- text and control sequences through xterm's data stream;
- binary input sequences;
- terminal column and row changes;
- pointer sequences emitted by xterm when Cinder enables mouse reporting;
- focus through the terminal viewport.

The guest writes ANSI output through `bridge.onOutput`. The host does not interpret or redraw the Cinder widget tree.

## Unicode

The browser runtime uses the same Cinder text and cell model as native terminals. Browser tests cover mixed Cyrillic, Arabic, emoji, and combining-mark input through the bridge.

Rendering still depends on terminal cell-width rules and the active browser font. Wide glyphs and grapheme clusters are owned by Cinder's buffer model rather than split into JavaScript UI elements.

## Adding an adapter

Create an executable Dart file whose name matches the generated example slug:

```text
docs-site/browser-adapters/example-slug.dart
```

The adapter must:

- render with Cinder;
- preserve the original example's intent;
- state its capability boundary in the UI;
- dispose timers and controllers;
- compile with `dart compile js`;
- remain deterministic in a static GitHub Pages deployment.

Do not modify generated manifests or bundles. Run:

```bash
cd docs-site
npm run prepare:site
npm run build
npm run test:routes
npm run test:browser
```

## Hosting

The production site uses a static Next.js export. GitHub project Pages sets:

```text
NEXT_PUBLIC_BASE_PATH=/cinder
NEXT_PUBLIC_SITE_ORIGIN=https://eukalpia.github.io
```

Bundle paths, source assets, iframes, the sitemap, and documentation links must all respect the same base path. A custom domain can use an empty base path without changing route structure.
