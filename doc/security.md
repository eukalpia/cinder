# Terminal security

Terminal applications process text from files, subprocesses, remote hosts, logs,
version-control tools, language servers, and network services. That text must be
treated as active input: terminal control sequences can change terminal state,
write to the clipboard, forge links, hide content, or make displayed output look
like trusted application UI.

Cinder separates terminal control generated by the framework from untrusted text
rendered by an application.

## Trust boundary

Use `TerminalText.safe` for text that did not originate in the local rendering
code:

```dart
Text(TerminalText.safe(processOutput));
Text(TerminalText.safe(remoteLogLine));
Text(TerminalText.safe(markdownCodeBlock));
```

Use `TerminalText.trusted` only when a sequence is intentionally generated by
framework or application infrastructure:

```dart
terminal.write(TerminalText.trusted(EscapeCodes.hideCursor));
```

`trusted` is deliberately explicit and performs no transformation. Its purpose
is to make reviews and security audits able to identify every call site that
permits terminal control.

## Sanitized input

`TerminalText.safe` removes:

- CSI/ANSI control sequences, including styling and cursor movement;
- OSC sequences, including title changes, hyperlinks, and OSC 52 clipboard writes;
- DCS, SOS, PM, and APC strings, including Sixel and Kitty-style payloads;
- C0 and C1 control characters other than optionally preserved tabs/newlines;
- bidirectional overrides and isolates that can visually reorder source text;
- selected zero-width formatting characters commonly used to conceal content.

Carriage returns are converted to newlines by default. This prevents progress
output from rewriting earlier content when it is displayed as a log.

Sanitization is not a shell-escaping API. Do not interpolate sanitized text into
shell commands, SQL, file paths, URLs, or process arguments without applying the
rules required by that destination.

## Threat model

### In scope

Cinder's terminal-output boundary is designed to prevent untrusted text from:

- changing the window title or icon name;
- moving the cursor or entering the alternate screen;
- modifying scroll regions or terminal dimensions;
- reading from or writing to the clipboard through terminal protocols;
- emitting deceptive OSC 8 hyperlinks;
- injecting Sixel, Kitty graphics, or arbitrary device-control payloads;
- changing foreground/background attributes to conceal text;
- corrupting terminal state after an error or interrupted operation;
- visually reordering text with bidirectional control characters.

### Application responsibilities

Applications remain responsible for:

- authenticating SSH and network connections;
- validating filesystem paths and archive extraction targets;
- separating command arguments instead of constructing shell strings;
- permission checks before destructive operations;
- redacting credentials, access tokens, and personal data from logs;
- bounding memory, image dimensions, decompression, and remote payload sizes;
- deciding whether OSC 52 clipboard integration is appropriate for the session.

### Trusted control paths

Low-level APIs such as `Terminal.write`, inline-image rendering, and PTY
forwarding may intentionally carry escape sequences. Keep these paths narrow.
Do not pass arbitrary process output into them. Render process output through a
widget after sanitization, or use an explicit terminal-emulator component that
implements its own parser and policy.

## Cancellation and cleanup

Security and reliability share the same lifecycle boundary. Interrupted work
must not leave a terminal in raw mode, retain SSH/SFTP channels, or keep reading
from a subprocess after its screen has closed.

Use `CinderTaskScope` for cooperative asynchronous cancellation and
`CinderResourceScope` for deterministic cleanup:

```dart
final tasks = CinderTaskScope();
final resources = CinderResourceScope();

final task = tasks.run((token) async {
  while (!token.isCancelled) {
    token.throwIfCancelled();
    await readNextBatch();
  }
});

@override
void dispose() {
  unawaited(tasks.dispose('widget disposed'));
  unawaited(resources.dispose());
  super.dispose();
}
```

Cleanup callbacks run in reverse registration order and continue after an
individual failure. Task disposal requests cancellation and waits for operations
to finish their cleanup paths.

## Testing requirements

Security-sensitive changes should include tests for:

- incomplete and fragmented CSI/OSC/DCS sequences;
- BEL and ST string terminators;
- OSC 52 and OSC 8 payloads;
- cursor movement, title changes, and alternate-screen commands;
- C1 equivalents of common escape sequences;
- combining marks, ZWJ emoji, wide CJK graphemes, and variation selectors;
- bidirectional overrides and isolates;
- cancellation during I/O, hashing, traversal, and subprocess execution;
- cleanup after exceptions and repeated cancellation.

Fuzz inputs should be bounded and deterministic in CI. Keep a regression corpus
for every sequence that previously bypassed sanitization or corrupted terminal
state.

## Reporting

Do not publish exploit details for an unpatched vulnerability in a public issue.
Contact the maintainers privately with a minimal reproduction, affected versions,
and the terminal environments used to verify the issue.
