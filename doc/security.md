# Terminal output security

Cinder treats every string that can originate outside the framework as hostile terminal input. LLM responses, shell output, Git data, file contents, server responses, Markdown, logs, plugin output, and user-provided labels can contain terminal control sequences even when they look like ordinary text in a debugger.

## Threat model

An attacker may control any text rendered inside a trusted Cinder application. Their goal may be to:

- execute arbitrary ANSI/CSI cursor and erase commands;
- write or read clipboard data through OSC 52;
- create malicious OSC 8 hyperlinks;
- change the terminal/window title;
- request terminal resize;
- reposition the cursor and draw a fake approval dialog;
- enter or leave the alternate screen;
- inject Kitty graphics, iTerm2 images, Sixel, DCS, APC, PM, or OSC payloads;
- hide or reorder text through foreground/background SGR or Unicode bidi controls;
- corrupt terminal modes so the shell remains in raw, mouse, paste, keyboard, scroll-region, or synchronized-output state after failure;
- consume excessive CPU/memory with pathological streams.

## Trust boundary

Use:

```dart
TerminalText.safe(untrustedOutput)
TerminalText.trusted(frameworkGeneratedDisplayText)
```

`TerminalText.safe` neutralizes controls before layout and paint. `TerminalText.trusted` validates that framework-generated **display text** contains no terminal controls. It is not an escape hatch for ANSI. Framework commands must be produced by typed `EscapeCodes`, image encoders, or backend presentation APIs.

Cinder also sanitizes at `RenderText`, rich paragraphs, Markdown parsing, `TerminalCanvas.drawText`, `Buffer.setString`, mutable `Cell` writes, differential encoding, and full-frame encoding. A custom render object that forgets to use `TerminalText.safe` still cannot place raw ESC/C1 commands in the physical frame.

## Sanitization policy

- LF is preserved.
- CRLF becomes LF; lone CR becomes the visible control picture `‚êç`.
- tabs become spaces.
- C0 controls become Unicode control pictures such as `‚êõ` and `‚êá`.
- DEL becomes `‚ê°`.
- C1 controls become visible `‚ü¶U+00XX‚üß` labels.
- bidi overrides/isolates and selected invisible format controls become visible named labels.
- emoji ZWJ and ordinary combining sequences remain valid graphemes.
- raw ANSI/OSC/DCS/APC/PM/Sixel/Kitty sequences therefore render as inert visible text.

The policy favors evidence over silent stripping: logs show that a control sequence was present without executing it.

## Terminal metadata and native image protocols

Window title and icon APIs sanitize and bound OSC metadata before wrapping it in framework-generated commands. Clipboard writes encode payloads as base64. Native image output is accepted only when `TerminalImagePayloadValidator` verifies that the entire payload is one matching Kitty, iTerm2, or Sixel sequence generated by the framework; protocol confusion and appended terminal commands are rejected.

## Embedded terminal exception

`TerminalXterm` is a terminal emulator boundary. PTY output is parsed into an internal screen model rather than forwarded to the host terminal. Host-side callbacks for title, clipboard, images, and other side effects must remain disabled or explicitly permission-gated. The outer Cinder canvas still sanitizes every grapheme emitted by the emulator.

## Fake approvals and visual spoofing

Sanitization blocks cursor movement, erase commands, SGR color changes, alternate-screen switching, and bidi overrides from untrusted strings. Applications must additionally make approval surfaces structurally distinct:

- approval data and action buttons are trusted widgets, not Markdown from the model;
- command, path, risk, and scope values use `TerminalText.safe`;
- keyboard focus and mouse hit regions come from the trusted widget tree;
- destructive actions require a real callback from the approval widget;
- text that merely looks like `[Approve]` is never actionable.

## Terminal-state recovery

Fatal cleanup attempts each operation independently: end synchronized output, reset scroll margins and styles, disable mouse modes, disable bracketed paste, pop Kitty keyboard/modifyOtherKeys modes, restore colors, show the cursor, leave alternate screen, repeat private-mode cleanup after the buffer switch, flush, and disable raw mode. Cleanup is idempotent and continues even when an individual backend write throws.

## Testing

The security suite covers ANSI, OSC 52, OSC 8, title, resize, cursor/erase, alternate-screen, Kitty, Sixel/DCS, bidi controls, malformed UTF-16, Unicode graphemes, raw cell injection, image protocol validation, OSC metadata boundaries, failure-injected terminal recovery, and deterministic fuzzing over both text and low-level cell input.

Security regression requirements:

- sanitized output never contains ESC or C1 CSI/OSC bytes;
- trusted display text rejects controls;
- no untrusted string reaches backend write APIs directly;
- sanitizer fuzz tests run in normal CI;
- terminal cleanup tests inject failures at every presentation stage.
